# 要件定義書:音声入力型 取引先情報管理ウェブアプリ

作成日:2026年7月24日
版数:v0.2(確認事項に暫定判断を反映。§7参照。いずれも変更可)

---

## 1. 背景と目的

営業活動や商談の後、取引先に関する情報(会話内容、担当者の特徴、次のアクションなど)をキーボードで入力するのは手間がかかり、記録が後回しになりがちである。本システムは、利用者が**口頭で話した内容をAIが自動で要約・整理**し、名刺写真とあわせて取引先ごとに蓄積できるウェブアプリケーションを構築することを目的とする。

取引先情報という機密性の高いデータを扱うため、**セキュリティを最優先**に設計する。

## 2. 利用者・利用環境

- 主な利用者:本人のみ(暫定決定)。ただしデータモデルは全レコードに所有ユーザーIDを持たせ、将来の少人数チーム拡張を妨げない設計とする
- 利用端末:スマートフォン(外出先での音声入力・名刺撮影)、PC(閲覧・編集)
- 利用形態:ウェブブラウザ(レスポンシブ対応)。ネイティブアプリは初期スコープ外

## 3. 機能要件

### 3.1 音声入力・自動要約(コア機能)

| 項目 | 内容 |
|---|---|
| 音声入力 | ブラウザのマイクから録音。話し終えたら停止ボタンで確定 |
| 文字起こし | 音声認識APIでテキスト化(日本語対応必須) |
| AI要約 | 文字起こし結果をLLMで構造化。「会社名/担当者/商談内容/次のアクション/メモ」等の項目に自動整理 |
| 確認・修正 | 要約結果を保存前に画面で確認・手動修正できる |
| 紐付け | 既存の取引先を自動判定して紐付け。新規の場合は取引先レコードを新規作成 |
| 元データ | 文字起こし全文も保存(要約だけでは拾えない情報の参照用)。音声ファイル自体は**文字起こし完了後に即時破棄し保存しない**(暫定決定。セキュリティ優先) |

### 3.2 名刺管理

| 項目 | 内容 |
|---|---|
| 撮影・アップロード | スマホカメラで撮影、または画像ファイルをアップロード |
| 保存 | 取引先・担当者に紐付けて保存 |
| OCR(推奨) | 名刺画像から会社名・氏名・電話・メールを自動抽出し、取引先情報に反映(手動修正可) |
| 表示 | 取引先詳細画面で名刺画像を閲覧・拡大表示 |

### 3.3 取引先管理

- 取引先一覧(会社名・最終接触日で並び替え)
- 検索:会社名、担当者名、メモ内容の全文検索
- 取引先詳細:基本情報、名刺画像、時系列の商談メモ(音声要約の履歴)
- 手動でのテキスト入力・編集も可能(音声が使えない場面のため)
- データのエクスポート(CSV等)※バックアップ・退避用

### 3.4 認証・アカウント

- ログイン必須(未ログインでは一切のデータにアクセス不可)
- 多要素認証(MFA)対応
- 一定時間操作がない場合の自動ログアウト

## 4. 非機能要件

### 4.1 セキュリティ要件(最重要)

| 分類 | 要件 |
|---|---|
| 通信 | 全通信をTLS 1.2以上で暗号化(HTTPS強制、HSTS有効化) |
| 保存データ | データベースおよび名刺画像ストレージを保存時暗号化(AES-256相当) |
| 認証 | パスワードは十分な強度を要求しハッシュ化保存(bcrypt/argon2)。MFA(TOTPまたはパスキー)対応 |
| セッション | セッショントークンの安全な管理、アイドルタイムアウト、CSRF対策 |
| アクセス制御 | 利用者本人のデータのみアクセス可能。将来のチーム利用時はロールベース制御 |
| 外部AI API | 音声認識・要約に外部APIを使う場合、**入力データが学習に利用されない**契約・設定(API経由利用)であることを必須条件とする |
| 監査ログ | ログイン履歴、データの閲覧・変更・削除の記録 |
| 脆弱性対策 | OWASP Top 10への対応(SQLインジェクション、XSS等)。依存ライブラリの脆弱性スキャン |
| バックアップ | 暗号化された定期バックアップと復旧手順の整備 |
| データ削除 | 取引先単位・アカウント単位での完全削除機能(個人情報保護法対応) |

### 4.2 その他の非機能要件

- 性能:音声要約は録音停止から30秒以内に結果表示を目標
- 可用性:個人利用前提のため99%程度で十分(暫定決定。マネージドサービス利用で実質的にはこれを上回る見込み)
- 対応ブラウザ:Chrome / Safari / Edge の最新版
- 個人情報保護:名刺・商談情報は個人情報を含むため、プライバシーポリシーの整備と法令(個人情報保護法)準拠

## 5. 技術構成(案)

| レイヤ | 候補 |
|---|---|
| フロントエンド | React / Next.js(PWA化でスマホ利用を快適に) |
| バックエンド | Node.js または Python(FastAPI) |
| データベース | PostgreSQL(暗号化対応のマネージドDB) |
| ストレージ | S3等のオブジェクトストレージ(名刺画像、非公開バケット+署名付きURL) |
| 音声認識 | Whisper API 等 |
| 要約・構造化 | Claude API 等(APIデータは学習不使用) |
| OCR | クラウドOCR API または LLMのビジョン機能 |
| 認証 | マネージド認証サービス(Auth0 / Firebase Auth / Clerk 等)を採用。セキュリティ最優先の方針のため自前実装は不採用(暫定決定) |
| インフラ | DB・ストレージは国内(東京)リージョンを第一候補。音声認識・要約のAI APIは海外処理を許容する(学習不使用の契約・設定が条件。暫定決定) |

## 6. スコープ外(初期リリースでは対象外)

- ネイティブアプリ(iOS/Android)
- 複数人での同時編集・共有機能
- 外部CRM(Salesforce等)との連携
- リアルタイム文字起こし(録音後の一括処理で開始)

## 7. 確認事項と暫定判断

v0.1の確認事項に対し、以下の暫定判断を置いて先行して進める。**いずれも最終確定ではなく、変更があれば該当箇所を改訂する。**

| # | 確認事項 | 暫定判断 | 理由 | 変更した場合の影響 |
|---|---|---|---|---|
| 1 | 利用者範囲 | **本人のみ**。ただし全データに所有ユーザーIDを持たせる | 実装・運用コスト最小。ID設計だけ先に入れておけば後からチーム化しても作り直しにならない | チーム利用に変更→ロールベース制御・招待機能の追加(中規模の追加工数) |
| 2 | 音声原本の扱い | **文字起こし完了後に即時破棄** | 最も漏えいリスクの高いデータを持たない構成が「セキュリティ最優先」に合致。ストレージ暗号化・削除機能の対象も減る | 保存に変更→暗号化音声ストレージ・保持期間管理・削除機能の追加 |
| 3 | 既存リストの移行 | **初期リリースには含めない**(CSVインポートはv2候補) | コア体験(音声→要約→蓄積)を最短で動かすことを優先。エクスポートは初期から実装するため、後からの取込追加も容易 | 初期に必要と判明→CSV/Excel取込画面の追加(小〜中規模) |
| 4 | 予算・リリース時期 | **低コスト構成(無料枠・従量課金の下限)で段階リリース**。まず音声入力+取引先管理のMVP、次に名刺OCR | 予算未確定のうちは固定費を作らない。従量課金のAI APIは個人利用量なら月数百〜数千円規模 | 予算確定→リージョン・冗長化・監視の水準を再設定 |
| 5 | データ保存場所 | **DB・画像ストレージは国内(東京)リージョン。AI APIのみ海外処理を許容** | 保存データの所在は国内に確保しつつ、AI APIの選択肢とコストを確保。AI APIは「入力を学習に使わない」契約(API利用)を必須条件とする | 完全国内限定に変更→利用可能なAI APIが大幅に絞られ、精度・コストを再評価 |
